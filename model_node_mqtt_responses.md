# Model Node MQTT 响应状态字典

本文档总结了在 `ModelNode` (基于 C++ 的 Node-RED 节点后端) 中，所有通过 `server_->response` 处理并最终通过 MQTT / WebSocket 或 Node-RED 内部总线输出的 JSON 消息格式及其作用。

## 基础报文结构 (Envelope)

所有的响应和事件都遵循以下标准 JSON 格式外壳：

```json
{
  "type": "<消息类别枚举，如主动事件(EVT)为2，被动响应(RESP)为1>",
  "name": "<具体的事件名称或动作名称>",
  "code": "<状态码，如 MA_OK(0) 代表成功，非0代表错误>",
  "data": "<实际承载的负载数据，格式视 name 而变>"
}
```

---

## 1. 核心 AI 推理结果流 (`name: "invoke"`)

* **发送方式**：连续流式发送（主动上报）
* **触发时机**：每处理完一帧图像，AI 模型推理完成并提取结果后触发。
* **数据格式**：

```json
{
  "type": 2, 
  "name": "invoke",
  "code": 0,
  "data": {
    "count": 123,                 // 迄今为止成功处理的帧累计数
    "resolution": [1920, 1080],   // 模型处理时原始画面的宽高分辨率
    "perf": [[10, 35, 5]],        // 性能耗时(ms)：[预处理耗时, 推理耗时, 后处理耗时]
    "image": "base64_string...",  // (选填) 当前画面的 JPEG Base64 编码，仅 `debug` 模式开启时存在
    "labels": ["person", "car"],  // 当前画面中识别到的所有物体标签名称集合
    
    // 【根据加载的模型类型不同，以下结果字段四选一】
    "boxes": [[x, y, w, h, score, target_id]],       // 目标检测：边界框、置信度和目标ID
    "classes": [[score, target_id]],                 // 图像分类：分数和对应的类别ID
    "keypoints": [{"box": [...], "pts": [[x,y,z]]}], // 姿态检测：身体边界框及骨骼关键点坐标
    "segments": [{"box": [...], "contour": [x,y]}],  // 实例分割：多边形边缘轮廓点集
    
    // 【根据附加功能开关，可能出现以下增叠字段】
    // (仅 trace=true 时存在)
    "tracks": [1, 2],             // 追踪目标的全局唯一追踪ID
    // (仅 counting=true 时存在)
    "counts": {"in": 5, "out": 2},// 拌线越界计数的累计值
    "lines": [[x1, y1, x2, y2]]   // 画在画面上的拌线实际渲染坐标
  }
}
```
* **作用**：向 Node-RED 下游节点传递 AI 处理后的视觉结构化数据，用于触发报警、录像或前端 UI 渲染绘制。

---

## 2. 节点部署/初始化成功 (`name: "create"`)

* **发送方式**：单次发送（事件响应）
* **触发时机**：Node-RED 点击 Deploy 时，后端 `onCreate` 函数成功读取模型文件(`*.cvimodel`)和配置文件(`*.json`)后触发。
* **数据格式**：

```json
{
  "type": 1,
  "name": "create",
  "code": 0,
  "data": {
    "classes": ["person", "bicycle", "car", "motorcycle"] // 从 model.json 提取出的元数据字典
  }
}
```
* **作用**：通知系统与前端节点准备就绪，验证模型加载成功，并下发当前模型支持的分类名称字典。

---

## 3. 工作状态同步 (`name: "enabled"`)

* **发送方式**：状态变化时发送
* **触发时机**：
  1. 后端工作线程刚启动时，主动广播一次初始状态。
  2. 接收到底层控制命令要求挂起 (pause) 或唤醒 (resume) 时。
* **数据格式**：

```json
{
  "type": 1,
  "name": "enabled",
  "code": 0,
  "data": true   // true 为正在运行，false 为挂起休眠状态
}
```
* **作用**：使前端界面能同步展示后端的启停状态，用于开关状态指示灯的切换。

---

## 4. 动态更改 AI 运行参数 (`name: "config"`)

* **发送方式**：命令回执 (ACK)
* **触发时机**：当 Node-RED 接收到动态控制消息以修改算法阈值、置信度等参数时 (`onControl` 函数处理成功后)。
* **数据格式**：

```json
{
  "type": 1,
  "name": "config",
  "code": 0,
  "data": {
    "tscore": 0.5,    // 确认已生效的新置信度阈值
    "tiou": 0.45,     // 确认已生效的新 NMS 非极大值抑制阈值
    "debug": true     // 确认调试画面的开关状态
  }
}
```
* **作用**：向上游节点确认动态配置已在 NPU/引擎层生效。相当于“签收回执”。

---

## 5. 指令不支持错误 (`code: -3 / MA_ENOTSUP`)

* **发送方式**：错误提示
* **触发时机**：如果接收到未知的控制动作（未命中的 control 字符串）。
* **数据格式**：

```json
{
  "type": 1,
  "name": "<用户传入的未知命令名称>",
  "code": -3,
  "data": "Not supported"
}
```
* **作用**：防卡死机制。由于系统是异步通讯，向前端抛出此错误可使发送方知晓指令下发失败，中断等待。

## 附录：系统总体生命周期与管控指令 (Server 层)

在 `ModelNode` 等具体节点之上，整个工厂框架 (`NodeServer`) 也会通过 MQTT 监听 `sscma/v0/<client_id>/node/in/<node_id>` 主题，处理以下系统级的指令。

### 6. 请求创建/实例化节点 (`name: "create"`)

* **方向**：入站 (Node-RED -> C++ 后端)
* **触发时机**：在前端画布上配置好一个节点并点击 **Deploy (部署)** 时下发。
* **数据格式**：
```json
{
  "name": "create",
  "data": {
    "type": "camera",  // 或者是 "model", "save" 等，代表要在 C++ 后端实例化的类属
    "fps": 30,         // ... 以及其余节点的初始化静态配置项
    "width": 1920
  }
}
```
* **作用**：命令后端的 `NodeFactory` 动态 `new` 出对应的 C++ 对象，并在底层做好内存分配和硬件初始化。

### 7. 请求销毁/回收节点 (`name: "destroy"`)

* **方向**：双向确认 (Node-RED <-> C++ 后端)
* **触发时机**：当在前端画布删除某个节点，或更改节点参数导致需要“先删后建”重新部署时下发。
* **输入控制格式 (前端发出)**：
```json
{
  "name": "destroy",
  "data": ""
}
```
* **输出确认格式 (后端回复)**：
```json
{
  "type": 1,
  "name": "destroy",
  "code": 0,    // 返回 MA_OK 代表销毁成功
  "data": ""
}
```
* **作用**：让后端调用底层指针的析构函数，安全释放摄像头占用、关闭文件流及释放信箱内存。向前端回复确认报文以允许 UI 完全移除该模块。

### 8. 全局工厂清空 (`name: "clear"`)

* **方向**：双向确认 (Node-RED <-> C++ 后端)
* **触发时机**：一键停用整套工作流，或者重启服务时从顶层下发。
* **输入控制格式 (前端发出)**：
```json
{
  "name": "clear",
  "data": ""
}
```
* **输出确认格式 (后端回复)**：
```json
{
  "type": 1,
  "name": "clear",
  "code": 0,
  "data": ""
}
```
* **作用**：触发 `NodeFactory::clear()`，遍历销毁容器中记录的**所有**节点实体，并将画布硬件状态彻底清零。

### 9. 动态控制透传路由 (自定义 Action 名)

* **方向**：入站透传 (Node-RED -> C++ 后端节点的 `onControl`)
* **触发时机**：在 Node-RED 运行中，通过 Inject 节点主动连接到底层时（发出的普通消息）。
* **数据格式**：
```json
{
  "name": "<任意自定义动作，例如 light, preview, config 等>",
  "data": "<具体的 payload 数据载荷，如 { \"tscore\": 0.6 } >"
}
```
* **作用**：由于 `name` 不是 `create/destroy/clear/health`，Server 会把它当做业务指令，利用 Topic 里的 `node_id` 查找到对应的节点实例，并**原样透传**给该实例的 `onControl()` 函数做业务执行（从而产生我们前面提到的回执响应）。

### 10. 健康探活监测 (`name: "health"`)

* **方向**：双向确认 (Node-RED <-> C++ 后端)
* **触发时机**：前端框架用于探测节点连通状态的心跳包。
* **数据格式 (前端发出)**：
```json
{
  "name": "health",
  "data": ""
}
```
* **作用**：C++ 服务端会立即返回同一 `name` 且 `code: 0` 的响应，用于确认底层服务进程、MQTT 连接以及事件处理线程池处于正常的就绪状态。

## 附加：Camera Node (摄像头核心外设) 专属状态响应

对于 `CameraNode` 这类直接与底层感光硬件、引脚交互的节点，除了前面提到的通用模型和系统回执外，还有其特有的硬件状态同步报文。这主要用于让前端 UI 精准展示“设备已连接”、“读取中”或响应控制硬件的指令。

### 1. 硬件挂载与参数初始化就绪 (`name: "create"`)

* **发送方式**：单次响应
* **触发时机**：配置部署阶段，后端执行 `initVideo()` 成功探活物理照相机（引脚和驱动通讯正常），并分配好底层分辨率后发出。前端用此判定为**硬件已连接就绪**。
* **数据格式**：
```json
{
  "type": 1,
  "name": "create",
  "code": 0,
  "data": {
    "width": 1920,   // 确认底层视频内存最终初始化的画面宽度
    "height": 1080,  // 确认底层视频内存最终初始化的画面高度
    "fps": 30        // 确认摄像头物理锁定的最大采样帧率
  }
}
```

### 2. 底层视频流启停状态同步 (`name: "enabled"`)

* **发送方式**：状态变化时发送
* **触发时机**：当硬件摄像头真正的产生第一帧并开始物理取流（触发 `CAMERA_INIT` 宏），或者被主动挂起休眠（触发 `CAMERA_DEINIT` 宏）时强制发送。前端节点依靠此报文点亮或熄灭代表“采流工作状态”的绿色指示灯。
* **数据格式**：
```json
{
  "type": 1,
  "name": "enabled",
  "code": 0,
  "data": true   // true 表示底层视频流已唤醒并源源不断接收入口帧，false 表示挂机/停止信号
}
```

### 3. 外设动态控制指轮回执 (`name: "light" / "preview"`)

* **发送方式**：指令回执 (ACK)
* **触发时机**：当系统运行时通过总线控制摄像头的具体外设硬件（如开关引脚补光灯）或业务（如临时开关降频预览图流）执行成功后。
* **数据格式**：
```json
{
  "type": 1,
  "name": "light",  // 或者是针对预览开关的 "preview"
  "code": 0,
  "data": {
    "light": 1      // 1确认补光灯引脚电平已拉高(亮起)，0代表关闭。"preview" 则在此返回其 bool 值
  }
}
```

### 4. 抽帧预览流总线投递 (`name: "sample"`)

* **发送方式**：受限事件流 (最多每 100ms 触发一次)
* **触发时机**：若未走带外 WebSocket 时，后端专门为 Node-RED 画布降频抽出的图片帧。（Model 叫 `invoke`，Camera 单纯给图的叫 `sample`）。
* **数据格式**：
```json
{
  "type": 2, 
  "name": "sample",
  "code": 0,
  "data": {
    "count": 45,                  // 已抽取并缓存发送的帧统计次数
    "image": "base64_string..."   // JPEG 图片的 Base64 数据串，供直接推给 img 标签展示
  }
}
```

## 附加：Model Node (AI 模型节点) 运行状态响应

与 Camera 类似，`ModelNode` 也具有自身的启动、加载和状态同步报文。这部分确保了前端 Node-RED 能感知到底层 AI 模型是否已被正确加载到 NPU 内存，以及当前是在持续推理阶段还是被挂起。

### 1. 物理模型挂载就绪与元数据解析 (`name: "create"`)

* **发送方式**：单次响应
* **触发时机**：部署阶段 `onCreate()` 函数成功将指定的 `.cvimodel` 文件加载到底层 NPU 引擎，并且成功挂载附属的 `.json` 标签配置时发出。
* **数据格式**：
```json
{
  "type": 1,
  "name": "create",
  "code": 0,
  "data": {
    "classes": ["person", "car", "dog"], // 如果模型附带 json，会返回它提取到的标签分类数组
    ...    // 这里会原样透传 model.json 里存储的关于该模型的其它所有元数据(info_)
  }
}
```

### 2. 模型推理启停状态同步 (`name: "enabled"`)

* **发送方式**：状态变化时发送
* **触发时机**：
  1. 后端的消费者线程 (`threadEntry`) 一旦成功启动并拉起循环，立刻主动发一次初始状态。
  2. 运行时收到前端传来的 `msg.payload = {"control": "enabled", "data": false/true}` 动态控制指令时触发，起到“修改并原路确认”的作用。
* **数据格式**：
```json
{
  "type": 1,
  "name": "enabled",
  "code": 0,
  "data": true   // true 表示模型正在不断 fetch() 获取图像并进行 AI 推理，false 表示在 while 循环中空转待机
}
```

### 3. AI 算法动态配置修改回执 (`name: "config"`)

* **发送方式**：指令回执 (ACK)
* **触发时机**：前端发送特定的管控指令临时修改 AI 推理结果的敏感度或显示选项时（进入 `onControl` 的 config 逻辑），返回确认内容。
* **数据格式**：
```json
{
  "type": 1,
  "name": "config",
  "code": 0,
  "data": {          // 这里的 data 会直接回显收到的那些配置改动项，表示修改成功
    "tscore": 0.65,  // NPU置信度阈值设为0.65
    "tiou": 0.40,    // NMS去重合并阈值
    "trace": true,   // 确认开启了骨干目标追踪机制
    "counting": true // 确认开启了越界计数功能
  }
}
```
